## 1. 已核实事实 / Verified Facts

> 全部事实**当前实测**自仓库真源；命令可复跑。

  `store_daily` / `talent_showcase` / `training_material`

### 1.2 生成配方层 / generation recipe layer 全是 domain_general（通用领域层）
- 真源：[knowledge_serving/views/generation_recipe_view.csv](knowledge_serving/views/generation_recipe_view.csv)

## 3. 最小可上线品牌知识包 / Minimal Viable Brand Pack

### P0 必交 / P0 Required

#### P0-1 · brand_voice（品牌嗓音）
- 影响：全 18 类内容输出口吻；不交则退化为 `domain_general` 默认口吻，品牌不可识别。
- 子项：
  - `tone_constraints_json`（语气约束 JSON）：可接受 / 不可接受的语气边界
  - `forbidden_words`（禁词清单）：品牌不允许使用的词
  - `signature_phrases`（标志性表达）：品牌常用语 / 口号
  - `output_structure_json`（输出结构）：文案的展开顺序、段落骨架

#### P0-2 · founder_persona（创始人画像）
- 影响：`founder_ip` 内容直接依赖；其他 17 类隐性引用以借审美锚点。
- 子项：
  - `origin_story`（创始人故事 / 品牌起源）
  - `value_anchors`（价值观锚点 / 不可妥协的底线）
  - `aesthetic_preferences`（审美偏好 / 风格关键词）
  - `signature_quotes`（签名短语 / 创始人常说的句子）

#### P0-3 · 品牌禁词 / 红线表达
- 影响：guardrail（护栏 / 静态规则校验）节点输入；缺则拦不住品牌不接受的表达。
- 形态：可嵌入 `brand_voice.forbidden_words`，也可单独维护。

#### P0-4 · business_brief（业务事实简报）数据通路
- 影响：每次内容生成必须提供商品 / 门店 / 活动 / 价格 / 库存 / 渠道等真实业务事实；缺失时 `fallback_status=blocked_missing_business_brief`，**系统拒生成最终成稿**（不许 LLM 编造商品事实）。
- 形态：不是品牌静态知识，而是**业务系统集成接口** —— 第二品牌的 ERP / OMS / 门店 POS 等真实业务数据流要能在生成时通过 API 输入。
- 校验：generation_recipe_view 的 `business_brief_schema_id` 字段定义了每个 content_type 的必填字段；第二品牌必须能持续提供这些字段。

### P1 看业务 / P1 Conditional

#### P1-1 · team_persona_overlay（团队角色画像）
- 影响：`role_work_vlog` / `store_daily` / `process_trace` 等涉及具体岗位的内容。
- 触发条件：**真实有此岗位才交**。
  - 有线下门店 → `store_associate`（门店导购）
  - 有设计师团队 → `designer`
  - 有打版 / 裁缝 → `sample_maker`
  - 纯线上品牌（无门店、无自研工艺）→ **可不交**，相关 content_type 退化用 `domain_general` 默认角色画像
- 红线：**不许虚构岗位**。第二品牌无门店却交 `store_associate` overlay = 内容造假。

#### P1-2 · content_type_overlay（内容类型专属覆盖）
- 影响：让某个 content_type 在第二品牌下表现出独特口吻。
- 策略：**只对 2-3 个招牌内容类型先做**；其余 15+ 类先走 `domain_general` 模板，跑通后再迭代补充。
- 笛语参照：仅为 `founder_ip` / `process_trace` / `store_daily` 3 类做了 overlay，其余 15 类用默认模板。

## 4. 证据与抽取要求 / Evidence & Extraction Rules

> 第二品牌**不能**只交一份口头品牌设定文档；必须按本仓 Phase 1 抽取纪律走完整 CandidatePack（候选知识包）流程。

### 4.1 每条品牌知识必须有
- `source_md`（原文 markdown 路径）：原始素材出处
- `evidence_quote`（证据原文引用）：能直接支撑该 knowledge_assertion（知识断言）的原文片段
- `evidence_id`（证据 ID）：稳定可追溯的 ID（按 [extract-9tables skill](.claude/skills/extract-9tables/) ID 生成规则）

### 4.2 4 闸 / 4 Gates 判定
所有第二品牌 CandidatePack 必须过 4 闸：
- Gate 1：knowledge_assertion（知识断言）能否写出具体且可执行
- Gate 2：success_pattern / flip_pattern（成功模式 / 翻转模式）能否反推
- Gate 3：evidence_quote 能否支撑 knowledge_assertion
- Gate 4：brand_layer 归属判定（按 CLAUDE.md §多租户隔离三步判定）

不过闸 → 进 `unprocessable_register`，**不许编空话凑数**。

### 4.3 brand_layer 归属判定（严格按 [CLAUDE.md §1 三步判定](CLAUDE.md)）
1. 是否落在 `domain_general` 范围（门店纪律 / 培训 / 陈列 / 面料 / 工艺 / 接客 / 库存 / 商品属性 / Schema 元规则）？是 → `domain_general`，结束。
2. 否，是否含**具体品牌调性或创始人画像**内容（非抽象规则）？是 → 进步骤 3；否 → `needs_review`。
3. 品牌归属：是第二品牌 → `brand_<新品牌 id>`；混合 → `needs_review`。

**反偷换警告**：不许把"门店运营规则形式可抽象"等同于"内容可去笛语化"；门店纪律 / 培训 / 陈列等仍是 `domain_general`。

## 6. 风险提醒 / Risk Reminders

### 6.1 禁止合成 brand_b 假绿
- 反模式：在 staging tenant_scope_registry.csv 写一行假的 `tenant_brand_b` + `brand_b` 数据 → 跑 Group C → 假装 KS-PROD-002 done。
- 危害：污染 staging 真源；下次真品牌上线时与假 brand_b 冲突；E8 drift normalization 反模式（让数据合理化漂移）。
- 真红线已写入 [task_cards/KS-PROD-002.md](task_cards/KS-PROD-002.md) blocked_reason 与 [task_cards/corrections/KS-FIX-24.md](task_cards/corrections/KS-FIX-24.md) unblock_when。

### 6.2 禁止 LLM 参与治理裁决
按 plan §6 step 2/3（2026-05-12 用户裁决 input-first / no-LLM）：
- 禁止 LLM 做 `intent_classification`（意图分类） / `content_type_routing`（内容类型路由） / `brand_layer` 推断 / `fallback` 裁决 / `merge` 裁决 / `evidence` 编造。
- 第二品牌的 brand_voice / founder_persona / overlay 选择**全部走确定性规则**：用户显式输入 + canonical csv 映射 + tenant_scope_resolver 推断；LLM 只在 `n7_llm_generation` 节点消费已组好的 context_bundle 写文案，不接管任何治理判断。

### 6.3 禁止把 domain_general 内容复制成 brand_x 专属
按 CLAUDE.md §3 反偷换警告：
- 门店纪律 / 培训规则 / 陈列搭配 / 接客判断 / 面料工艺 / 库存替代 / 商品属性 / Schema 元规则 → 永远是 `domain_general`，**不许**为了"显示第二品牌有内容"而复制成 `brand_<新品牌 id>` 专属。
- 判别问句：**"这条规则换到别的品牌是不是必须重写？"** 必须重写 → `brand_<name>`；不必重写 → `domain_general`。

### 6.4 不许跨层反向引用
- `domain_general` 行**永远**不许引用任何 `brand_<name>` 行（包括 brand_faye 和未来的第二品牌）。
- `brand_<name>` 行可以引用 `domain_general` 行（这是设计正向）。

## 7. 给品牌方的材料清单（大白话版）

> 把 §3 翻译成第二品牌客户的市场 / 运营团队能直接读懂的提交清单。新手友好。

### A. 必交（不交无法正式上线）

1. **一份"品牌说话风格手册"**（对应 brand_voice）
   - 我们品牌说话的口气是什么？（活泼 / 克制 / 文艺 / 专业……）
   - 哪些词我们坚决不用？（举例：性价比、真香、闭眼入、贵妇感……）
   - 哪些是我们的标志性短语？（品牌口号、常用比喻、信号词）
   - 一篇典型文案的结构是什么样的？（先说什么、后说什么、最后落什么）
   - **提交形态**：1-2 个 markdown 文档 + 5-10 篇品牌已发的真实样本文案（带链接出处）。

2. **一份"创始人画像档案"**（对应 founder_persona）
   - 创始人为什么做这个品牌？关键转折点是什么？
   - 创始人在选材 / 审美 / 对客户态度上的底线是什么？
   - 创始人个人的审美偏好（颜色 / 质感 / 风格关键词）
   - 创始人常说的几句标志性的话（带原文出处，例如某次直播 / 某篇专访）
   - **提交形态**：1 份创始人访谈纪要 + 至少 3 段可引用的真实原话 + 出处（直播录屏 / 公众号文章 / 媒体采访等）。

3. **一份"品牌禁词与红线表达清单"**（对应 guardrail 输入）
   - 哪些表达品牌坚决不接受？（包括但不限于价格暗示、敏感词、低俗联想等）
   - **提交形态**：1 份禁词表 CSV / markdown。

4. **业务系统对接（business_brief 数据通路）**
   - 每次发内容前，要能告诉系统："这次要讲哪个 SKU、什么品类、当季什么活动、什么价格区间、哪些渠道发"。
   - 这部分不是"知识"，是**业务系统集成**：第二品牌要提供 API / 数据通路，让生成时能拿到这些真实业务事实。
   - **缺这一项 → 系统主动拒生成商品类内容**（不许 LLM 编造商品事实）。

### B. 选交（有就交，没有就退化用通用模板）

5. **"团队岗位画像"**（对应 team_persona_overlay）
   - 你们品牌真实有哪些岗位？（导购、设计师、打版师、加盟商、店长……）
   - 每个岗位的工作场景、口吻特点、典型场景。
   - **红线**：没有这个岗位就不要交。例如纯线上品牌没门店 → 不要交"门店导购"画像。

6. **"招牌内容类型专属口吻"**（对应 content_type_overlay）
   - 在 18 类内容里，你们品牌最重视哪 2-3 类？（例如主打"创始人 IP" / "幽默搞笑" / "知识科普"）
   - 这几类内容专属的表达约束是什么？
   - 其他 15+ 类先走通用模板，跑通再迭代。

### C. 不需要交（这些已经是平台共享资产）

- 怎么写一篇 OOTD（每日穿搭）—— 模板平台有
- 怎么写一篇产品测评 —— 模板平台有
- 门店接待流程的通用规则 —— 平台有
- 面料工艺的通用知识 —— 平台有
- 商品属性 / 库存替代逻辑 —— 平台有
- 9 张表 / 4 闸的结构 —— 平台有

### D. 提交后的真实工程流程（让品牌方知道审核会发生什么）

1. 我们会把品牌方交的所有素材，**拆成 CandidatePack（候选知识包）**。
2. 每条素材必须**有原文出处**（不能只交"创始人价值观是……"这种总结，必须配上"创始人在 2025 年 X 月 Y 日某访谈里原话说……"）。
3. 拆好的 CandidatePack 要过**4 个自检闸门**（能不能写出具体规则 / 能不能反推成功失败 / 原文能不能支撑结论 / 该归到通用层还是品牌专属层）。
4. 过闸的内容才进 9 张表，并打上 `brand_<新品牌 id>` 标签。
5. 然后我们才会把这套品牌数据上传到生产服务器，并和现有的笛语数据做跨租户隔离测试 —— 必须做到：你的品牌看不到笛语的数据，笛语也看不到你的数据。
6. 隔离测试通过后，KS-PROD-002 任务卡才能正式标记完成。
