# 笛语知识库工作机制说明

> **定位**：[笛语知识库建设路线图.md](笛语知识库建设路线图.md) 的姊妹篇——路线图讲"什么时候做什么"，本文讲"做出来后怎么工作"。
> **读者**：所有想理解笛语 KB 价值的人，不需要技术背景。

---

## Slogan · 一句话讲清这个系统

> **让 LLM 在每次生成时同时拥有：行业判断、品牌气质、真人细节、可拍动作、审稿红线。**

五个齿轮咬合，LLM 才会从"写文案"变成"替笛语组织一次真实表达"。

---

## 一句话版

笛语知识库让 LLM **先知道"这是什么内容"、再知道"这类内容高级在哪里"、再知道"笛语不能怎么说、应该像谁说"、再拿到真实事件和人物细节，最后用规则和反模式把废话、AI 味、假高级感拦掉**，于是吐出来的就不只是文案，而是有品牌灵魂、有现场动作、有镜头证据的脚本。

---

## 一、阶段作用

### M0.5 · 先建水管

这一阶段不是让 LLM 变聪明，而是**防止以后变脏**。

仓库里强调先建 `raw_material_register`、授权字段、`evidence_maturity`、brand overlay 立法。它的作用是规定：

- 什么素材能进库，什么不能进
- 什么只能当候选，什么能变成稳定知识
- 什么是笛语专属，什么是行业通用
- 真实访谈、朋友圈、客户反馈不能直接喂给 LLM，必须先**清洗、授权、抽取、过闸**

> **这一层解决的是"可信来源"和"长期可维护"。**

### M1-M2 · 把笛语的声音和真人装进去

- **M1**：定创始人、BrandTone、Persona 情绪光谱
- **M2**：采样衣师、店长、导购、加盟店主四个真人岗位

这时知识库开始出现"血肉"：口头禅、真实冲突、一次高光、一次低谷、真实判断动作、和创始人的关系。

LLM 以后写脚本时，不再只能写：

> ❌ "我们坚持品质。"

而能写成：

> ✅ "样衣师把那件衣服拎起来看了三秒，说这道缝今天不能过。不是不好看，是顾客坐下来会先替它尴尬。"

**这就是血肉层的价值。**

### M3 · 首次生成，开始形成闭环

M3 是关键拐点。路线图里写得很明确：用 `founder_ip` / `process_trace` / `store_daily` 各生成 3-5 条，然后做**五维盲测**。

这一步开始让知识库和 LLM 发生真实互动：

1. LLM 生成内容
2. 人判断"像不像安映华""像不像笛语""能不能发""像不像真实门店"
3. 失败内容回查对应 raw / pack 缺陷
4. 知识被标记为 `stale` / `refresh` / `retire`

也就是说，**LLM 不是一次性被训练好，而是在内容实战里暴露知识库哪里空、哪里假、哪里太模板**。

### M4-M6 · 从会写，到稳定产出

- **M4** 扩领域通用护城河：服装零售、商品工艺、一线岗位、翻车案例
- **M5** 加流程溯源、商品故事、平台合规
- **M6** 做 stale、deprecation、跨源验证、第二品牌评估

到这里，知识库就不只是"笛语素材库"，而变成**三层协同系统**：

| 层 | 内容 | 作用 |
|---|---|---|
| `domain_general` | 服装零售、搭配、面料、陈列、门店、内容方法的通用判断力 | 行业护城河 |
| `brand_faye` | 笛语的声音、审美、禁忌、创始人和团队血肉 | 品牌灵魂 |
| `runtime assets / play cards` | 可以直接被 LLM 调用的镜头模板、动作模板、话术模板、玩法卡 | 可执行素材 |

## 二、生成时的真实调用链

一次好的脚本生成，大概会这样发生：

### 1. 先识别任务
比如用户要"拍一条新品测评视频"，系统先判断这是 `product_review`，还是 `process_trace`，还是 `store_daily`，避免一上来就乱写。

### 2. 调 L1 概念层
L1 告诉 LLM：**这类内容的北极星是什么，什么算好，什么算烂**。

例如产品测评不是"显瘦百搭"，而是"把顾客顾虑变成现场可见证据"。

### 3. 调 L2 玩法层
L2 给出**可拍玩法**。

比如仓库里的 play card 有：
- "把顾客当场说出的顾虑变成三十秒可见测试"
- "用三件同类衣服的声音和触感做一场法庭裁决"

> **这一步决定内容会不会"眼前一亮"。**

### 4. 调 L3 执行资产
L3 给**镜头、动作、话术**。

比如：
- "先摸后讲面料"
- "原相机自然光不拉腿不美颜"
- "顾客提出扎热透勒时先识别真实体感维度"

> **这一步让脚本从"会说"变成"能拍"。**

### 5. 叠加 brand_faye
这一步让内容**变成笛语，而不是任何女装品牌**。

它会控制语气、禁忌、审美表达、创始人/店长/导购的真实说话方式。

比如不把高级感讲成"贵妇感"，而讲成"松弛、真实、好接近"。

### 6. 叠加真实 raw 血肉
把某次样衣修改、某个顾客反馈、某个门店现场、某个失败事件放进脚本。

> **这一步让内容从"对"变成"动人"。**

### 7. 反模式审稿
最后用规则拦掉 AI 味、空泛口播、假高级、无证据卖点、过度模板化。

---

### 所以最终吐出的不是

> ❌ "这件衣服采用优质面料，适合通勤和约会。"

### 而是

> ✅ "镜头先别拍正面。让她坐下、抬手、拿包、弯腰捡手机。
> 这件衣服值不值得讲，不看它站着漂不漂亮，看它有没有在这些动作里露怯。"

**这才是笛语知识库真正想养出来的东西。**

## 三、核心机制

最关键的不是 RAG，而是**分层协作**：

| 层 | 管什么 | 一句话本质 |
|---|---|---|
| L1 | 判断 | 这条内容**为什么成立** |
| L2 | 玩法 | 这条内容**怎么抓人** |
| L3 | 执行 | 这条内容**怎么拍、怎么说、怎么剪** |
| brand_faye | 灵魂 | 为什么它**是笛语** |
| raw 血肉层 | 真实 | 为什么它**不像 AI 编的** |
| 评估闭环 | 进化 | 为什么**下一条会更好** |

**所以，笛语知识库最终让 LLM 惊艳的方式，不是给它更多资料，而是让它在每次生成时同时拥有：行业判断、品牌气质、真人细节、可拍动作、审稿红线。**

这几个齿轮咬住以后，LLM 才会从"写文案"变成"替笛语组织一次真实表达"。

## 四、5 点架构校准（理解时机别错位）

上面的工作机制是**完成态**——大约 M5/M6 之后才能五层齿轮全咬合。中间几个时间点容易误解，这里点清楚：

### 校准 1：L3 执行层目前是"规划态"，不是"已就位"

调用链里"调 L3 执行资产 → 镜头/动作/话术"这一步——是路线图 **M5 之后**才会真填的能力。当前仓库 L3 = 24 个 pack，W12.B 推迟没抽。

**前 3 个月（Stage 0/1）LLM 调用链里 L3 是空的**，只能靠 L2 玩法层 + raw 血肉直接出脚本。

### 校准 2：反模式审稿现在还不是"运行时引擎"

仓库里 `success_pattern + flip_pattern` 是**抽取期**的成对要求（保证规则有正反两面），**不是生成期的运行时校验器**。

要把 flip_pattern 变成"自动拦 AI 味/假高级"的审稿引擎，需要 MVP 引擎那边写一段校验代码——**这一步在本仓库红线外**。

> 现在我们准备的是"审稿用的素材"，不是"审稿的发动机"。

### 校准 3：检索调用链不是本仓库做的

第二节描述的"先识别任务 → 调 L1 → 调 L2 → 调 L3 → 叠加 brand_faye → 叠加 raw → 反模式审稿"——这是一段 **pipeline 实现**。

**本仓库不做这段 pipeline**，是 MVP/diyu-infra-05 引擎做。本仓库只**供货**：9 表 + brand_faye + raw register + ContentBlueprint 模板。

### 校准 4：evidence_maturity 三级是去单源水分的关键

单源 brand_faye 喂出来的"通用判断"其实是**笛语认知伪装的通用**。`single_source_candidate → multi_scene_validated → cross_brand_validated` 三级把这层水分挤出去。

**没有它，Stage 3/4 跨品牌迁移走不通**——通用规则永远停留在"笛语通用"，不是真通用。

### 校准 5：stale / deprecation 是长期防腐剂

评估闭环管"进化"，stale 检测管"防腐"。6 个月没新证据的规则自动标 stale；不然 12 个月后 KB 里会堆一堆"曾经对、现在过期"的规则**污染生成**。

## 五、仓库 vs MVP 分工（最容易搞糊的边界）

> **一句话：仓库管"配料 + 配方"，MVP 管"灶台 + 厨师"。**

| 工位 | 本仓库（20-血肉-2F种子）| MVP 仓库（diyu-infra-05）|
|---|---|---|
| 知识真源（规则 + 血肉）| ✅ 主产 | — |
| ContentBlueprint 模板 | ⏳ 部分（M4 玩法卡）| ✅ 主产 6 个 |
| 检索引擎（向量 + 关系查询）| — | ✅ 主产 |
| 提示词组装 pipeline | — | ✅ 主产 |
| LLM 调用 + 结构化切回 | — | ✅ 主产 |
| 反模式校验闸（运行时）| — | ✅ 主产 |
| 平台适配（抖音/小红书/视频号）| — | ✅ 主产 |
| 5 维盲测 | ✅ 流程立法 + 评估表 | ✅ 内容生成支持 |
| 转换器（yaml→Pydantic）| ⏳ 待用户授权 | — |

### 配料表 → 厨师 的桥梁

本仓库产物：

```
9 表（CSV）+ brand_faye 包（YAML）+ raw_material_register（CSV）+ ContentBlueprint 模板
```

要进入 MVP 厨房，必须有一个**转换器**：把 yaml/csv 转成 MVP 的 Pydantic JSON。这个转换器没写，**两边就咬合不上**。

转换器是个 1-2 天的脚本活，但需要用户明确授权（CLAUDE.md 红线"不做物理分库 / 多方案转换脚本"）。
